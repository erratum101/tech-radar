## 1. Введение и цели

Система предназначена для визуализации технологического ландшафта команды/организации в формате радара (квадранты и кольца), чтобы:

- быстро фиксировать и обсуждать статус технологий;
- стандартизировать коммуникацию между командами;
- снизить время на подготовку презентационных материалов;
- поддерживать обмен состоянием радара без сложной серверной инфраструктуры.

### 1.1 Бизнес-цели


| Цель                               | Описание                                                                                       |
| ---------------------------------- | ---------------------------------------------------------------------------------------------- |
| Единый источник статуса технологий | Сформировать единый и понятный артефакт для технического управления и продуктовых команд.      |
| Ускорение архитектурных обсуждений | Сократить время подготовки и согласования технологических решений за счет визуального формата. |
| Упрощение обмена между командами   | Передавать конфигурацию радара по ссылке без ручной пересборки данных.                         |
| Снижение порога входа              | Обеспечить работу без обязательной регистрации и бэкенда в базовом сценарии.                   |
| Поддержка регулярных ревизий       | Использовать инструмент как основу квартальных/спринтовых ревью технологического стека.        |

## 3. Границы и окружение системы

### 3.1 Бизнес-контекст

Внешние участники:

1. **Инженерные команды** — формируют и обновляют содержимое радара.
2. **Архитектурный комитет/техлиды** — используют радар для принятия решений.
3. **Руководители направлений** — получают агрегированную картину технологического ландшафта.
4. **Внутренние потребители ссылки** — открывают опубликованный радар и используют его как read-only/read-mostly артефакт.

### 3.2 Технический контекст


| Партнер                                      | Описание взаимодействия                      | Вход                  | Выход                                       |
| -------------------------------------------- | -------------------------------------------- | --------------------- | ------------------------------------------- |
| Browser APIs                                 | `localStorage`, `Clipboard API`, `URL API`   | Действия пользователя | Сохраненное состояние, скопированная ссылка |

| Пользовательский URL                         | Импорт состояния радара из `radar` параметра | Сжатая строка         | Восстановленные настройки и элементы        |

## 5. Компонентный состав

### 5.1 Верхнеуровневая компонентная схема

Система состоит из клиентского SPA-приложения со следующими блоками:

- `App` как оркестратор состояния и маршрутов отображения;
- UI-компоненты представления радара (`HomeScreenView`, `FullRadarView`, `QuadrantDetailView`);
- подсистема редактирования (`SettingsOverlay`, `ElementEditor`, dropdown-компоненты);
- утилиты сериализации/сжатия и импорта XLSX;
- клиентские механизмы хранения/обмена (`localStorage`, `URL hash/query`, clipboard).

### 5.2 Описание компонентов


| Компонент             | Назначение                                           | Входящие коммуникации            | Исходящие коммуникации                    |
| --------------------- | ---------------------------------------------------- | -------------------------------- | ----------------------------------------- |
| `App`                 | Центральная координация состояния, загрузки, шаринга | UI события, URL, localStorage    | Рендер дочерних экранов, update state     |
| `SettingsOverlay`     | Изменение конфигурации радара и элементов            | Команды пользователя             | Обновление `settings`, `allRadarItems`    |
| `FullRadarView`       | Полный вид радара с интерактивными квадрантами       | Данные департамента + координаты | Выбор квадранта                           |
| `QuadrantDetailView`  | Детальный просмотр выбранного квадранта              | Фильтрованные данные             | Отображение списка и blips                |
| `HomeScreenView`      | Стартовый экран и пустое состояние                   | Текущее состояние                | Навигация пользователя                    |
| Утилиты share payload | Сериализация/десериализация, компрессия              | `settings`, `items`, `uiTheme`   | URL-ready строка / восстановленный объект |
| XLSX loader           | Импорт демо-данных                                   | Файл `radar_data.xlsx`           | Массив нормализованных элементов          |


### 5.3 Основные интерфейсы

Для текущей версии ключевые интерфейсы — это внутренние контракты frontend и browser APIs:

1. `loadTestData()` — загрузка тестового XLSX.
2. `fetchAndProcessTestData()` — чтение/парсинг workbook.
3. `encodeSharePayload(settings, items, theme)` — формирование короткой строки для URL.
4. `decodeSharePayload(encoded)` — восстановление состояния из URL.
5. `buildShareUrlWithPayload(encoded)` — сборка итоговой share-ссылки.
6. `loadSharedRadarFromUrl()` — загрузка состояния из `radar` в hash/query.
7. `copyTextToClipboard(text)` — запись ссылки в буфер.
8. `handleReset()` — сброс к пустому состоянию.
9. `localStorage.getItem/setItem` — хранение тем и сохраненных стилей.

### 5.3.1 Описание ключевого интерфейса Share Radar

**Назначение:** передача полного состояния радара через URL без backend.  
**Ключевые этапы:** build payload -> compress/encode -> append to URL -> decode on open.

```yaml
interface: ShareRadar
input:
  settings: object
  allRadarItems: array
  uiTheme: "light|dark"
output:
  url: string
constraints:
  - payload must be versioned (v field)
  - backward compatibility for older payload versions
  - must support base path normalization for deployed static hosts
errors:
  - invalid payload format
  - decode/decompress failure
fallback:
  - user notification + keep current local state unchanged
```

### 5.4 Детализация ключевых сущностей

1. **RadarSettings**

`radarTitle`, `logo`, `departments[]`, `quadrants[]`, `rings[]`, `colors`.

1. **RadarItem**

`name`, `department`, `quadrant`, `ring`, `owners[]`.

1. **SavedStylePreset**

`id`, `name`, `settings`, `isDefault`.

1. **SharePayload (v5)**

`v`, `s` (settings), `u` (theme), `p` (dictionary of owners), `t` (compact items tuple).

## 6. Динамическое представление

### 6.1 Перечень сценариев


| Сценарий                        | Компоненты                                                                     |
| ------------------------------- | ------------------------------------------------------------------------------ |
| Загрузка тестовых данных        | `App` -> `fetchAndProcessTestData` -> `xlsx` -> state update                   |
| Просмотр радара по департаменту | `App` -> `FullRadarView` -> координаты -> UI                                   |
| Просмотр деталей квадранта      | `FullRadarView` -> `QuadrantDetailView`                                        |
| Публикация радара по ссылке     | `App` -> `encodeSharePayload` -> `buildShareUrlWithPayload` -> `Clipboard API` |
| Открытие радара по ссылке       | URL -> `decodeSharePayload` -> `normalizeLogoForDeployment` -> state update    |


### 6.2 Иллюстрации (sequence UML, текстом)

#### 6.2.1 Сценарий: публикация радара по ссылке

1. Пользователь нажимает «Поделиться текущим радаром».
2. `App` формирует payload из `settings`, `allRadarItems`, `uiTheme`.
3. Payload компрессируется (`gzip+msgpack` / `gzip+json` / `lz`), выбирается кратчайший вариант.
4. Формируется URL с `#radar=...`.
5. Выполняется попытка копирования в буфер обмена.
6. При успехе показывается toast, при ошибке — prompt и alert.

#### 6.2.2 Сценарий: открытие shared-ссылки

1. Приложение читает `radar` из hash/query.
2. Выбирается путь декодирования по префиксу/формату.
3. Восстанавливается объект состояния и нормализуется путь логотипа под текущий `BASE_URL`.
4. Обновляются `settings`, `allRadarItems`, `allEmployees`, `uiTheme`.
5. Пользователю показывается уведомление об успешной загрузке.

## 7. Схема развёртывания

- Тип: статический SPA deploy.  
- Артефакт: `vite build` -> папка `dist`.  
- Хостинг: любой статический HTTP(S) (GitHub Pages, S3 static site, nginx).  
- Требования:
  - корректно настроенный `BASE_URL` при деплое в подпапку;
  - HTTPS для корректной работы `Clipboard API` в современных браузерах.

## 8. Архитектурные и дизайн-решения (ADR)

1. **ADR-001:** Выбрана SPA без backend для MVP.
2. **ADR-002:** Состояние обмена передается через URL вместо серверного хранения.
3. **ADR-003:** Для компактности ссылки используется комбинированная стратегия сжатия и сериализации.
4. **ADR-004:** Схема share payload версионируется для обратной совместимости.
5. **ADR-005:** Настройки пользователя сохраняются в `localStorage` для мгновенного восстановления.

## 9. Требования к атрибутам качества

### 9.1 Дерево атрибутов качества

- **Удобство использования**
  - Простой путь «создать -> настроить -> поделиться».
  - Информативные уведомления при ошибках.
- **Производительность**
  - Быстрый рендер основных экранов.
  - Приемлемое время обработки share payload.
- **Надежность**
  - Устойчивость к невалидному URL payload.
  - Безопасный fallback без потери текущего состояния.
- **Изменчивость**
  - Расширяемость структуры `settings`/`items`.
  - Локальность изменений в UI-компонентах.

### 9.2 Сценарии проверки атрибутов качества

1. **Performance:** при 1000 элементах радара время генерации share-ссылки не превышает 1 сек на типовом ноутбуке.
2. **Reliability:** при поврежденном `radar` параметре приложение не падает и показывает сообщение об ошибке.
3. **Usability:** новый пользователь выполняет сценарий импорта и публикации ссылки за <= 5 минут.
4. **Maintainability:** добавление нового поля у `RadarItem` требует изменений не более чем в 3-4 файлах.

## 10. Риски и технический долг

### 10.1 Основные риски реализации решения


| Риск                                                                                                                       | Митигация                                                                                                                                                                                                                                                  |
| -------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Превышение лимита длины URL при шаринге «всё состояние в ссылке» — ссылка не откроется или обрежется в почте/мессенджере.  | В продакшене по умолчанию использовать серверное сохранение снимка радара и короткую ссылку (`srv:<id>`); длинный URL оставить как fallback. Ограничить размер клиентского payload, предлагать экспорт в файл для очень больших радаров.                   |
| Рост нагрузки на API шаринга (много одновременных `POST`/`GET`) или исчерпание диска при хранении в файле.                 | Rate limiting по IP/ключу, лимит размера тела запроса, ротация/архивация старых снимков, мониторинг диска и метрик API. При росте нагрузки — вынести хранилище в БД или объектное хранилище, горизонтально масштабировать инстансы API за балансировщиком. |
| Недоступность API шаринга, потеря или порча файла `radars.json` на сервере — ссылки `srv:` перестают работать.             | Health-check и алерты, регулярные бэкапы каталога данных, при сбое — понятное сообщение пользователю; на клиенте сохранён fallback на длинную ссылку. Технический долг: реплицируемое хранилище (БД + бэкапы).                                             |
| Отсутствие аутентификации: злоупотребление созданием снимков, мусорные/вредоносные payload, отсутствие «владельца» радара. | Ограничение частоты запросов, валидация схемы payload на сервере, опционально капча или одноразовый токен для `POST`. Технический долг: вход через OIDC/OAuth2 и квоты на пользователя.                                                                    |
| Данные только в `localStorage` — очистка браузера или другой ПК = потеря пресетов и черновиков.                            | Явное предупреждение в UI, экспорт/импорт пресетов в файл. Технический долг: облачное сохранение «моих радаров» после внедрения учётных записей.                                                                                                           |
| Неверная конфигурация деплоя (`BASE_URL`, прокси `/api`) — ломаются ассеты, шаринг или загрузка по ссылке.                 | Чек-лист деплоя в документации, smoke-тест после релиза (открытие страницы, шаринг, открытие `srv:`-ссылки), мониторинг HTTP-ошибок на `/api`.                                                                                                             |
| Регрессии в логике сериализации/десериализации share payload и в импорте XLSX — «сломанные» ссылки и данные.               | Автотесты round-trip для кодеков, фикстуры для старых версий формата, тест на образце XLSX; при изменении формата — сохранение обратной совместимости или миграция версий.                                                                                 |


### 10.2 Технический долг

- Совместное редактирование в реальном времени не входит в текущий объём; при необходимости — отдельный этап (WebSocket + модель конфликтов или CRDT).  
- Защита от DDoS на статику и API — на уровне хостинга/WAF и лимитов; детальная проработка по мере выхода в публичный интернет с высоким трафиком.

## 11. Словарь

- **Tech Radar** — визуальная модель статуса технологий по квадрантам и кольцам.  
- **Quadrant** — тематическая область (например, инструменты, платформы).  
- **Ring** — степень зрелости/рекомендации (например, Adopt/Trial/Assess/Hold).  
- **Share Payload** — сериализованное и сжатое состояние радара в URL.  
- **MVP** — минимально жизнеспособная версия продукта.  
- **SPA** — одностраничное веб-приложение.
